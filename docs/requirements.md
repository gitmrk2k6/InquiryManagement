# 要件定義書 — 問い合わせ管理アプリ（InquiryManagement）

## 1. 概要・目的

### 1.1 背景

企業がユーザーから受け取る「問い合わせ」は、製品・サービスの改善や顧客満足度を支える重要な接点である。
一方で多くの現場では、共有メールアドレスやスプレッドシートに依存した運用が続いており、対応漏れ・二重対応・進捗の不可視化といった課題が発生しやすい。

本プロジェクトでは、**BtoC 顧客窓口**を想定した問い合わせ管理 Web アプリケーションを開発する。
顧客側は公開フォームから問い合わせを送信でき、社内のカスタマーサポート担当は管理画面から一元的に対応状況を管理できる。

### 1.2 目的

- 顧客が **24時間いつでも** 問い合わせを送信できる窓口を提供する
- 受け付けた問い合わせを **取りこぼさず可視化** する
- 対応状況を **ステータス（未対応／対応中／完了）** で管理し、対応漏れ・二重対応を防ぐ
- フェーズ2 では、社内コメント・担当者割当・メール通知により **チーム対応** を効率化する

### 1.3 本書の位置づけ

本ドキュメントは RaiseTech AI エンジニアコース **初級編最終課題** として作成する。
要件定義の **目次（ハブ）** として機能し、機能要件・画面設計・データベース設計などの詳細は、後続で別ドキュメントへ分割管理する。

機能要件・画面設計・DB 設計はフェーズ2 まで描き切る一方、**実装着手は MVP（フェーズ1）に限定する** ことを本書の前提とする。

## 2. 解決する課題

### 2.1 想定する現状（Before）

- 顧客からの問い合わせを共有メール（例: `support@example.com`）で受けている
- 対応状況は Excel／スプレッドシートやメールのフラグで管理しており、担当者間で共有が難しい
- 「誰が対応中か」「未対応のまま何日経過したか」が見えづらく、対応漏れ・二重対応が発生する

### 2.2 課題と解決方法

| # | 誰の課題 | 現状の課題 | 本アプリでの解決方法 | 関連機能 |
| --- | --- | --- | --- | --- |
| C-01 | 顧客 | 営業時間外に問い合わせ手段がない／電話が繋がらない | 24時間受付可能な公開フォーム | F-01 |
| C-02 | 顧客 | 自分の問い合わせが届いたか分からず不安 | 送信完了画面で受付番号を提示（フェーズ2: 受付完了メール） | F-02, F-09 |
| C-03 | 一次対応者 | メール一覧では対応状況が分からない | ステータス付きの問い合わせ一覧画面 | F-04, F-05 |
| C-04 | 一次対応者 | 誰が対応中か共有できず二重対応が起きる | 担当者割当機能（フェーズ2） | F-08 |
| C-05 | 責任者 | 対応漏れ・SLA 超過に気づけない | ステータスフィルタ・経過日数表示 | F-06 |
| C-06 | 全員 | 対応経緯が個人メールに散在し引き継ぎが困難 | 社内コメント（対応履歴）でアプリ内に集約（フェーズ2） | F-07 |

## 3. 想定ユーザー

### 3.1 ペルソナ

| ペルソナ | 役割 | IT リテラシー | 本アプリでの主な行動 | 代表的な期待 |
| --- | --- | --- | --- | --- |
| 一般顧客（送信者） | 商品／サービスの BtoC 利用者 | 低〜中（スマホで Web フォームが使える程度） | 公開フォームから問い合わせを送信／送信完了を確認 | 簡単に・短時間で送れる、送信した実感が得られる |
| 一次対応担当（カスタマーサポート） | サポートチームメンバー | 中（業務 SaaS を日常利用） | 一覧で新着確認 → 詳細閲覧 → ステータス更新 → 完了化 | 新着が一目で分かる、対応漏れが起きない |
| サポート責任者（マネージャ） | チームリーダー／管理者 | 中〜高 | 全件のステータス把握、長期滞留案件の検出、（フェーズ2）担当割当 | 全体状況を一画面で把握、長期滞留を検出できる |

### 3.2 認証スコープ

本アプリは **2サイドプロダクト** であり、利用者によって認証要否が異なる。

- **一般顧客**: 認証なし（未ログイン状態で公開フォームにアクセス）
- **一次対応担当 / 責任者**: **管理者ログイン（JWT）** を必須とし、管理画面にアクセスする

MVP では一次対応担当と責任者の権限差は設けず、両者を「管理者」フラットな1ロールとして扱う。
管理者間の権限分離（role 管理）は **フェーズ2** で導入を検討する（[F-11](#5-機能要件概要)）。

## 4. ユーザーストーリー（主要シナリオ）

「**〜として、〜したい。なぜなら〜だから。**」形式で主要なシナリオを示す。詳細なユースケースは [機能要件書](functional-requirements.md) で定義する。

| ID | ストーリー | 対応機能 |
| --- | --- | --- |
| US-01 | **顧客** として、24時間いつでも問い合わせを送信したい。なぜなら営業時間内に電話する余裕がないから。 | F-01 |
| US-02 | **顧客** として、送信が完了したことを確認したい。なぜなら届いていないと不安だから。 | F-02 |
| US-03 | **一次対応者** として、未対応の問い合わせ件数を一目で把握したい。なぜなら対応漏れを防ぎたいから。 | F-04, F-06 |
| US-04 | **一次対応者** として、対応開始時にステータスを「対応中」に変更したい。なぜなら他のメンバーとの二重対応を防ぎたいから。 | F-05 |
| US-05 | **責任者** として、長期間「未対応」のままの問い合わせを検出したい。なぜなら SLA 超過を防ぎたいから。 | F-06 |
| US-06 | **一次対応者** として、対応経緯を社内メモとして残したい。なぜなら別担当に引き継ぐ場面があるから。（フェーズ2） | F-07 |

## 5. 機能要件（概要）

MVP（フェーズ1）と拡張（フェーズ2）に分けて整理する。
詳細仕様・バリデーション・ユースケースは [機能要件書](functional-requirements.md) を参照。

| 区分 | ID | 機能名 | 概要 | 想定ユーザー | フェーズ |
| --- | --- | --- | --- | --- | --- |
| 公開 | F-01 | 問い合わせ送信フォーム | 氏名・メール・件名・本文を入力して送信 | 顧客 | MVP |
| 公開 | F-02 | 送信完了画面 | 受付番号と案内文を表示 | 顧客 | MVP |
| 認証 | F-03 | 管理者ログイン／ログアウト | メール+パスワードで JWT を発行／破棄 | 管理者 | MVP |
| 管理 | F-04 | 問い合わせ一覧 | 全問い合わせをページング表示 | 管理者 | MVP |
| 管理 | F-05 | 問い合わせ詳細・ステータス変更 | 詳細閲覧と「未対応／対応中／完了」の切替 | 管理者 | MVP |
| 管理 | F-06 | 一覧の絞り込み・ソート | ステータス別フィルタ、受信日時ソート、経過日数表示 | 管理者 | MVP |
| 管理 | F-07 | 対応履歴（社内コメント） | 内部メモ投稿、時系列表示 | 管理者 | フェーズ2 |
| 管理 | F-08 | 担当者割当 | 問い合わせに担当管理者をアサイン | 管理者 | フェーズ2 |
| 通知 | F-09 | 受付完了メール（顧客向け） | 送信時に顧客へ自動メール送信 | 顧客 | フェーズ2 |
| 通知 | F-10 | ステータス変更通知 | 完了時に顧客へ自動メール送信 | 顧客 | フェーズ2 |
| 認証 | F-11 | 管理者ユーザー管理（role） | 管理者の追加・削除・役割付与 | 責任者 | フェーズ2 |

## 6. 画面設計（概要）

本アプリは **公開側** と **管理側** の二層構成とする。

**公開側（未認証）**
- 問い合わせフォーム画面
- 送信完了画面

**管理側（要ログイン）**
- 管理者ログイン画面
- 問い合わせ一覧画面（フィルタ／ソート／ページング）
- 問い合わせ詳細画面（ステータス変更／フェーズ2: コメント・担当割当）
- 管理者ユーザー管理画面（フェーズ2）

画面一覧・ワイヤーフレーム・画面遷移図は [画面設計書](screen-design.md) を参照。

## 7. データベース設計（概要）

主要エンティティは以下の通り。詳細な ER 図・テーブル定義・インデックス・制約は [データベース設計書](database-design.md) を参照。

- `inquiry` — 問い合わせ本体（受付番号、氏名、メール、件名、本文、ステータス、受信日時 など）
- `admin_user` — 管理者アカウント（メール、パスワードハッシュ、role ※フェーズ2 など）
- `inquiry_comment` — 社内コメント（フェーズ2）
- `inquiry_assignment` または `inquiry.assignee_id` — 担当者割当（フェーズ2）

> 受付番号体系（例: `INQ-2026-00042`）は MVP では `inquiry.id` の流用で十分とし、独立した採番ルールはフェーズ2 で検討する。

## 8. 非機能要件

| カテゴリ | 要件 |
| --- | --- |
| 性能 | 通常操作（一覧表示・ステータス変更等）が **2 秒以内** に応答すること。公開フォームの送信は **3 秒以内**（顧客の離脱防止） |
| セキュリティ | パスワードはハッシュ化（bcrypt 等）して保存。JWT は HttpOnly Cookie 推奨。XSS／CSRF／SQL Injection 対策を施す |
| 個人情報保護 | 氏名・メール等の個人情報を扱うため **HTTPS 必須**。問い合わせ本文はログに出力しない。削除・匿名化方針を別途定義する |
| スパム対策 | 公開フォームは未認証のため、**reCAPTCHA／honeypot／IP 単位レートリミット** のいずれかを必須とする |
| 可用性 | 学習用途のため高可用構成は対象外（単一 EC2 + RDS で動作すれば可） |
| 保守性 | NestJS の Module / Controller / Service / Repository 構造で層分割する |
| 互換性 | モダンブラウザ最新版（Chrome / Firefox / Edge / Safari）で動作。**スマホ閲覧対応必須**（顧客はスマホから送信する想定） |

## 10. スコープ外（今回は実施しない）

MVP・フェーズ2 のいずれにおいても、以下は本プロジェクトの対象外とする。

- 顧客側のマイページ・ログイン（自分の問い合わせ履歴閲覧）
- リアルタイム通知（WebSocket 等での即時反映）
- 問い合わせ本文への **ファイル添付** 機能
- メールでの双方向やり取り（メール返信を取り込む機能）
- チャットボット／FAQ 連携／自動回答
- 多言語対応
- スマホネイティブアプリ
- 監視・アラート・SLA メトリクスの自動収集
- バックアップ・DR（災害復旧）設計
- SSO（Google / Microsoft 連携での管理者ログイン）
